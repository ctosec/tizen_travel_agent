1. DOCUMENT OVERVIEW
================================================================================

  This document describes the software architecture, module design, data flow,
  and implementation details of the Tizen Travel Agent application — an AI-
  powered travel booking system designed for Samsung Tizen Smart TVs.

  The application allows users to:
    - Browse destination attractions with AI-generated descriptions
    - Generate AI-powered day-by-day travel itineraries
    - Search and book flights via Amadeus GDS
    - Search and book hotels via Amadeus GDS
    - Pay via QR-to-Phone payment flow (Toss Payments)
    - Navigate entirely with a TV remote (D-pad + Enter)

  2.3  Communication Patterns
  ────────────────────────────

  Frontend → Backend:
    - REST API over HTTP (JSON)
    - Base URL configured via VITE_API_URL environment variable
    - All endpoints under /api prefix
    - No authentication (guest mode)

  Backend → Frontend:
    - WebSocket (Socket.IO) at /payments namespace
    - Used exclusively for real-time payment status push

  Backend → External APIs:
    - Amadeus GDS: SDK-based (amadeus npm package)
    - Google Gemini: SDK-based (@google/generative-ai)
    - Google Places: REST API with API key header
    - Toss Payments: REST API with Basic auth

    │    Right: Payment panel (price + method + QR)

    │  Phone scans QR → pays → TV updates via WebSocket
    ▼
  (Payment Success shown inline on BookingPage)

  Back navigation: Backspace / XF86Back key returns to previous page

  3.6  API Client
  ────────────────

  Base URL: VITE_API_URL + '/api' (e.g., https://server.com/api)

  Endpoints Called:
    GET  /destination/{country}/{city}          → DestinationData
    POST /itinerary/generate                    → ItineraryDay[]
    POST /flights/search                        → FlightOffer[]
    POST /hotels/search                         → HotelOffer[]
    POST /payments/sessions                     → { orderId, sessionId }
    GET  /payments/sessions/{orderId}/status     → { status, data? }
    GET  /server-info                           → { baseUrl }

  WebSocket (socket.io-client):
    Namespace: /payments
    Events:
      emit 'subscribePayment' (orderId)       → Join room
      on   'paymentStatus' ({ status, data })  → Receive updates
      emit 'unsubscribePayment' (orderId)      → Leave room

  3.8  TV Viewport & Responsive Scaling
  ──────────────────────────────────────

  Design resolution: 1920 x 1080 (fixed)

  Scaling algorithm (main.tsx):
    scale = Math.min(window.innerWidth / 1920, window.innerHeight / 1080)
    root.transform = scale(${scale})
    root.marginLeft = (window.innerWidth - 1920 * scale) / 2
    root.marginTop = (window.innerHeight - 1080 * scale) / 2

  This provides:
    - Uniform scaling (no distortion) on any TV resolution
    - Letterbox centering when aspect ratio differs from 16:9
    - Gradient background fills letterbox area
    - Works on 720p, 1080p, 4K TVs

  CSS (index.html):
    - body: 100vw x 100vh, overflow hidden, cursor none
    - #root: 1920px x 1080px, transform-origin top left
    - Scrollbars hidden globally (still scrollable internally)

  4.5  Service Details
  ─────────────────────

  4.5.1  AmadeusService
  .....................

  SDK: amadeus@9.1.0 (test or production environment)
  Initialization: Connects with clientId, clientSecret, hostname from env

  Methods:
    searchFlights(origin, destination, date, adults, class)
      → amadeus.shopping.flightOffersSearch.get()
      → Returns max 10 flights in KRW currency

    searchHotels(cityCode, checkIn, checkOut, adults)
      → Step 1: amadeus.referenceData.locations.hotels.byCity.get()
      → Step 2: amadeus.shopping.hotelOffersSearch.get() (top 20 hotels)
      → Returns offers with bestRateOnly in KRW

    bookFlight(flightOffer, travelers)
      → Step 1: amadeus.shopping.flightOffers.pricing.post() (get final price)
      → Step 2: amadeus.booking.flightOrders.post() (create order)
      → Extracts PNR from associatedRecords[0].reference
      → Includes ticketing agreement (24hr queue)

    bookHotel(offerId, guests, payments)
      → amadeus.booking.hotelOrders.post()
      → Maps guests to room associations

  4.5.2  GeminiService
  .....................

  Model: gemini-2.5-flash (Google Generative AI)

  Methods:
    generateText(prompt) → string
      Plain text generation

    generateJSON<T>(prompt) → T
      Generates text, strips markdown fences, parses JSON
      Used for structured data (itineraries, descriptions)

  4.5.3  PlacesService
  .....................

  API: Google Places API v1 (REST)

  Methods:
    textSearch(query, maxResults)
      → POST https://places.googleapis.com/v1/places:searchText
      → Headers: X-Goog-Api-Key, X-Goog-FieldMask
      → Language: Korean (ko)
      → Returns: placeId, name, address, location, photoRef, rating

    getPhotoUrl(photoRef, maxWidth)
      → Constructs Google Places photo URL
      → Used by photo proxy endpoint

  4.5.4  DestinationService
  .........................

  Combines Places + Gemini for rich destination data:
    1. Check 10-minute in-memory cache
    2. Search Google Places for top 5 attractions
    3. Generate Korean descriptions via Gemini AI
    4. Map photos to /api/places/photo/ proxy URLs
    5. Cache result for 10 minutes

  4.5.5  ItineraryService
  .......................

  Generates AI-powered travel itineraries:
    1. Check 10-minute in-memory cache (key: city-duration-date)
    2. Call Gemini to generate day-by-day schedule (3 activities/day)
    3. For each activity location, search Google Places for photos
    4. Set photo URLs to /api/places/photo/:ref?maxWidth=400
    5. Fallback: Pre-built Barcelona itinerary or generic activities
       - Timing: 10:00, 14:00, 19:00 (adjusted for Day 1 and last day)

  4.5.6  PaymentsService
  .......................

  Toss Payments confirmation:
    If SKIP_TOSS_PAYMENT=true (mock mode):
      Returns mock success data with calculated VAT
      Maps selectedMethod to Korean payment provider names
      Generates random approval number

    If SKIP_TOSS_PAYMENT=false (production):
      POST https://api.tosspayments.com/v1/payments/confirm
      Auth: Basic base64(secretKey + ':')
      Body: { paymentKey, orderId, amount }

  4.5.7  PaymentSessionsService
  .............................

  Manages PaymentSession entity lifecycle:
    createSession(data)  → Creates session with 30-min expiry
    getSession(orderId)  → Returns session or null
    getSessionStatus(orderId) → Checks expiry, returns status
    updateSessionStatus(orderId, status, paymentKey?, resultData?)
                         → Prevents re-updating SUCCESS sessions
    cleanExpiredSessions() → Batch-expire old PENDING sessions

  4.5.8  BookingsService
  ......................

  CRUD operations on Booking entity:
    createBooking(data) → Creates with payment & Amadeus data
    findAll()           → All bookings, ordered by createdAt DESC
    findById(id)        → Single booking
    findByIds(ids)      → Multiple bookings by ID array
    updateBookingStatus(id, status) → Status update


  4.6  WebSocket Gateway
  ───────────────────────

  Namespace: /payments
  CORS: origin: '*'

  Events:
    Connection:
      handleConnection(client)     → Logs client ID
      handleDisconnect(client)     → Logs client ID

  5.3  WebSocket Payment Status Monitoring
  ─────────────────────────────────────────

  Frontend Strategy: Dual monitoring (WebSocket + HTTP polling)

  WebSocket:
    1. Connect to ws://server:3000/payments (Socket.IO)
    2. Emit 'subscribePayment' with orderId
    3. Listen for 'paymentStatus' events
    4. Update paymentStore on status change

  HTTP Polling (fallback):
    1. Every 3 seconds: GET /payments/sessions/{orderId}/status
    2. If status changed from PENDING → update store
    3. Stop polling on SUCCESS or FAIL

  Both mechanisms run in parallel for reliability.
